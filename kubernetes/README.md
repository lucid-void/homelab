# Kubernetes — homelab-k8s

Parallel Kubernetes cluster running alongside the existing Docker Swarm homelab.
Full design rationale lives in [design/kubernetes.md](../design/kubernetes.md).

## Directory structure

```
kubernetes/
├── talos/
│   ├── talconfig.yaml          # node definitions — edit this
│   ├── talsecret.sops.yaml     # cluster PKI + tokens (SOPS-encrypted, generated once)
│   └── clusterconfig/          # gitignored — generated by talhelper
├── flux/
│   ├── pub-cert.pem            # Sealed Secrets public cert (safe to commit)
│   ├── flux-system/            # FluxCD bootstrap manifests (generated by flux bootstrap)
│   ├── infrastructure/
│   │   ├── controllers/        # operators: Sealed Secrets, CNPG, democratic-csi, Cilium
│   │   └── configs/            # StorageClasses, etc. (dependsOn: controllers)
│   └── apps/                   # applications (dependsOn: configs)
└── images/
    └── postgres-cnpg-immich/   # custom Dockerfile — built on GitHub Actions, pushed to GHCR
```

## Cluster overview

| Node | IP | Role |
|---|---|---|
| `k8s-cp-1` | 172.16.20.20 | Control plane + workloads (VIP holder when elected) |
| `k8s-cp-2` | 172.16.20.21 | Control plane + workloads |
| `k8s-cp-3` | 172.16.20.22 | Control plane + workloads |
| VIP | 172.16.20.19 | API server endpoint — floats between control planes |

All three nodes run both control-plane components (etcd, kube-apiserver, kube-scheduler, kube-controller-manager) and user workloads (`allowSchedulingOnControlPlanes: true` removes the NoSchedule taint). Losing any one node maintains etcd quorum (2 of 3) and cluster operation continues uninterrupted.

| Layer | Tool |
|---|---|
| OS | Talos Linux |
| CNI | Cilium (WireGuard encryption, Hubble) |
| CSI | democratic-csi — iSCSI (RWO) + NFS (RWX) from Synology |
| GitOps | FluxCD |
| Secrets | Sealed Secrets |
| Postgres | CloudNativePG operator |
| Ingress | Traefik |

## Prerequisites

Install tooling (one-time):

```bash
# Talos
brew install siderolabs/tap/talosctl
brew install talhelper

# Kubernetes
brew install kubectl helm fluxcd/tap/flux

# Sealed Secrets
brew install kubeseal
```

**Talos image — managed by talhelper:**

Extensions are declared in `talos/talconfig.yaml` under each node's `schematic.customization.systemExtensions` block.
Talhelper registers the schematic with factory.talos.dev automatically on `talhelper genconfig` and embeds the resulting image URL in the generated clusterconfig files.

The schematic ID used for Packer must match what talhelper resolves — after running `talhelper genconfig`, grep the generated config for the current ID:

```bash
grep "metal-installer" kubernetes/talos/clusterconfig/homelab-k8s-k8s-cp-1.yaml
```

Update `talos_schematic_id` in [`infra/packer/Talos/talos-base.pkr.hcl`](../infra/packer/Talos/talos-base.pkr.hcl) to match.

Packer downloads the ISO directly from factory.talos.dev during Phase 4 — no manual image upload needed.

## Phase 1 — Generate Talos cluster secrets

Run once. The output contains the cluster PKI and join tokens. Back it up — if lost, the cluster must be rebuilt from scratch.

```bash
cd kubernetes/talos/

talhelper gensecret > talsecret.sops.yaml
sops -e -i talsecret.sops.yaml
git add talsecret.sops.yaml && git commit -m "feat(talos): cluster secrets"
```

## Phase 4 — Build Talos template (Packer)

Packer creates a Proxmox VM template with the Talos ISO attached and the correct hardware spec.
No provisioning happens — Talos has no SSH or shell. The template is purely a hardware definition.

Before building, confirm `talos_version` and `talos_schematic_id` in
[`infra/packer/Talos/talos-base.pkr.hcl`](../infra/packer/Talos/talos-base.pkr.hcl) match the
versions in `talos/talconfig.yaml` and the schematic shown in the Prerequisites section.

```bash
# from repo root
just build-talos-template
```

This downloads the Talos ISO to Proxmox local storage and creates template VM `9001`.
Run once per Talos version upgrade.

## Phase 5 — Provision VMs (Tofu)

Tofu clones the Talos template and creates the three k8s VMs.

**Before running:** add static DHCP leases on the UDM SE for the fixed MAC addresses below.
Talos does not use cloud-init — IP addressing on first boot comes from DHCP.
The static lease must match the `ipAddress` in `talos/talconfig.yaml` so that `talhelper apply`
can reach each node at its expected address.

| Node | MAC | IP |
|---|---|---|
| `k8s-cp-1` | `BC:24:11:00:20:00` | `172.16.20.20` |
| `k8s-cp-2` | `BC:24:11:00:21:00` | `172.16.20.21` |
| `k8s-cp-3` | `BC:24:11:00:22:00` | `172.16.20.22` |

After the first `talhelper apply`, the static IP is written into the Talos machine config
and persists independently of DHCP. The DHCP reservation is only needed for first-boot reachability.

```bash
just plan
just apply
```

VMs boot into **maintenance mode** — Talos ISO starts, no config applied yet, API reachable on port 50000.

## Phase 6 — Apply Talos machine configs

Configs are generated into `talos/clusterconfig/` (Phase 3). Push each one to the node at its actual IP.
On first apply the node has no PKI — `--insecure` skips certificate verification.

If you set up the UDM SE DHCP reservations (Phase 5), the nodes boot at the expected IPs.
If not, check the UDM SE lease table for the actual IPs and substitute below.

```bash
talosctl apply-config \
  --nodes 172.16.20.20 --insecure \
  --file kubernetes/talos/clusterconfig/homelab-k8s-k8s-cp-1.yaml

talosctl apply-config \
  --nodes 172.16.20.21 --insecure \
  --file kubernetes/talos/clusterconfig/homelab-k8s-k8s-cp-2.yaml

talosctl apply-config \
  --nodes 172.16.20.22 --insecure \
  --file kubernetes/talos/clusterconfig/homelab-k8s-k8s-cp-3.yaml
```

Each node reboots after receiving its config. After reboot the static IP in the machine config is permanent — DHCP is no longer involved.

## Phase 7 — Bootstrap etcd

**Run once, on one node only.** The other two control planes discover the first and join automatically.
Re-running on an existing cluster corrupts etcd.

```bash
talosctl bootstrap \
  --nodes 172.16.20.20 \
  --endpoints 172.16.20.20 \
  --talosconfig kubernetes/talos/clusterconfig/talosconfig
```

Wait for all three control planes to form a healthy etcd cluster:

```bash
talosctl health \
  --nodes 172.16.20.20,172.16.20.21,172.16.20.22 \
  --endpoints 172.16.20.20 \
  --talosconfig kubernetes/talos/clusterconfig/talosconfig
```

## Phase 9 — Bootstrap Cilium + Sealed Secrets

A single helmfile command installs Cilium (CNI) and Sealed Secrets in dependency order.
Values live in `flux/infrastructure/controllers/cilium/helm-values.yml` — shared with the later Flux HelmRelease.

### New cluster (first time — no existing Sealed Secrets key)

Just run helmfile. The Sealed Secrets controller generates a fresh key on first start.
Seal all secrets in Phase 10 after helmfile completes.

```bash
helmfile --file kubernetes/bootstrap/helmfile.yml apply --skip-diff-on-install --suppress-diff

until kubectl wait --for=condition=Ready nodes --all --timeout=600s; do sleep 10; done
kubectl get nodes
```

### Restoring an existing cluster (Sealed Secrets key already backed up)

Restore the controller key **before** helmfile so the controller picks it up on start
and can decrypt all existing SealedSecrets in git without re-sealing anything.

```bash
# Decrypt the backed-up key from Synology and apply it
sops -d /mnt/backups/keys/sealed-secrets-key.sops.yaml | kubectl apply -f -
#kubectl apply -f main.key #if you're using existing secrets

# Then run helmfile — controller starts with the restored key
helmfile --file kubernetes/bootstrap/helmfile.yml apply --skip-diff-on-install --suppress-diff

until kubectl wait --for=condition=Ready nodes --all --timeout=600s; do sleep 10; done
kubectl get nodes
```

## Phase 10 — Seal secrets

Do this once after Sealed Secrets is running. The `.secret.yml` files are templates — fill them in, seal, then delete the plaintext.

```bash
# GitHub deploy key for Flux (generate a new SSH keypair, add public key to GitHub repo as deploy key)
ssh-keygen -t ed25519 -C "flux@homelab-k8s" -f /tmp/flux-deploy-key -N ""
# Add /tmp/flux-deploy-key.pub as a deploy key on https://github.com/lucid-void/Homelab

# Fill in the private key in the template
sed "s|<paste private key here>|$(cat /tmp/flux-deploy-key)|" \
  kubernetes/bootstrap/flux/github-deploy-key.secret.yml | \
kubeseal -f kubernetes/bootstrap/flux/github-deploy-key.secret.yml -w kubernetes/bootstrap/flux/github-deploy-key.sealed.yml
kubeseal -f kubernetes/flux/vars/cluster-secrets.yml -w kubernetes/flux/vars/cluster-secrets.sealed.yml

shred -u /tmp/flux-deploy-key
git add kubernetes/bootstrap/flux/github-deploy-key.sealed.yml \
        kubernetes/flux/vars/cluster-secrets.sealed.yml
git commit -m "feat(k8s): sealed bootstrap secrets"
git push
```

## Phase 13 — Back up the Sealed Secrets key

**Do this immediately after install and after any manual key rotation.**
If the cluster is rebuilt without this backup, all SealedSecrets in git are permanently unreadable.

```bash
kubectl get secret -n kube-system \
  -l sealedsecrets.bitnami.com/sealed-secrets-key \
  -o yaml > /tmp/ss-key.yaml

# encrypt with existing SOPS age key and store on Synology
sops -e /tmp/ss-key.yaml > /mnt/backups/keys/sealed-secrets-key.sops.yaml

shred -u /tmp/ss-key.yaml
```

Cache the public cert for offline sealing:

```bash
kubeseal --fetch-cert \
  --controller-name=sealed-secrets-controller \
  --controller-namespace=kube-system \
  > kubernetes/flux/pub-cert.pem

git add kubernetes/flux/pub-cert.pem
git commit -m "feat(k8s): sealed-secrets public cert"
git push
```

## Phase 14 — Seal and deploy secrets

Write plaintext → seal → commit → delete plaintext. Never reverse this order.

```bash
# write plaintext secret (example)
cat > /tmp/secret.yaml <<EOF
apiVersion: v1
kind: Secret
metadata:
  name: immich-db-credentials
  namespace: immich
type: Opaque
stringData:
  DB_PASSWORD: "changeme"
EOF

# seal it
kubeseal --cert kubernetes/flux/pub-cert.pem \
  --format yaml < /tmp/secret.yaml \
  > kubernetes/flux/apps/immich/sealed-db-credentials.yaml

shred -u /tmp/secret.yaml

git add kubernetes/flux/apps/immich/sealed-db-credentials.yaml
git commit -m "feat(immich): db credentials"
git push
```

FluxCD picks up the commit, the controller decrypts the SealedSecret, the `Secret` is created in the cluster.

### Upgrade Talos (version or schematic change)

The schematic (extensions) is defined in `talos/talconfig.yaml` under each node's `schematic:` block.
Talhelper registers it with factory.talos.dev and bakes the resulting image URL into the generated clusterconfig.

**Step 1 — update talconfig.yaml and regenerate**

```bash
# edit talos/talconfig.yaml (bump talosVersion, add/remove extensions, etc.)
cd kubernetes/talos
talhelper genconfig
```

**Step 2 — get the installer image URL for each node**

```bash
grep "metal-installer" clusterconfig/homelab-k8s-k8s-cp-1.yaml
# e.g. factory.talos.dev/metal-installer/<schematic-id>:v1.x.y
# use factory.talos.dev/installer/<schematic-id>:v1.x.y for the upgrade command (installer, not metal-installer)
```

**Step 3 — upgrade one node at a time**

Use `--drain=false` — the CNPG PodDisruptionBudget blocks eviction of the primary pod.
Upgrade nodes that are *not* holding the CNPG primary first where possible.

```bash
talosctl upgrade \
  --nodes 172.16.20.20 \
  --image factory.talos.dev/installer/<schematic-id>:<version> \
  --talosconfig kubernetes/talos/clusterconfig/talosconfig \
  --drain=false

# wait for the node to come back up, then repeat for .21 and .22
```

**Step 4 — apply config to deliver extension service configs**

The upgrade reboots the node with the new image, but `ExtensionServiceConfig` documents
(lldpd, netbird, etc.) must be pushed separately via `apply-config`:

```bash
talosctl apply-config \
  --nodes 172.16.20.20 \
  --file kubernetes/talos/clusterconfig/homelab-k8s-k8s-cp-1.yaml \
  --talosconfig kubernetes/talos/clusterconfig/talosconfig
```

If the node stalls at `BOOTING` waiting for extension services, check which services are blocked:

```bash
talosctl dmesg --nodes 172.16.20.20 --talosconfig kubernetes/talos/clusterconfig/talosconfig | tail -5
talosctl service ext-netbird --nodes 172.16.20.20 --talosconfig kubernetes/talos/clusterconfig/talosconfig
```

Update `talosVersion` in `talos/talconfig.yaml` after all nodes are upgraded.

## Recreate — One control plane (cluster still has quorum)

With 3 control planes, losing one keeps etcd quorum (2 of 3). The cluster continues running throughout.
No etcd snapshot restore is needed — the rebuilt node simply rejoins the live cluster.

```bash
# 1. reprovision via Tofu (or just re-apply to recreate the VM)
just apply

# 2. push Talos config to the replaced node (--insecure: no PKI on fresh VM)
talosctl apply-config \
  --nodes 172.16.20.21 --insecure \
  --file kubernetes/talos/clusterconfig/homelab-k8s-k8s-cp-2.yaml

# 3. node boots, receives config, contacts existing etcd, rejoins automatically
kubectl get nodes --watch   # node transitions NotReady → Ready
```

## Recreate — All control planes lost (etcd unrecoverable)

Only needed if all three control planes are lost simultaneously (full cluster wipe).

```bash
# 1. reprovision all three VMs via Tofu
just apply

# 2. push Talos config to all nodes
talosctl apply-config --nodes 172.16.20.20 --insecure \
  --file kubernetes/talos/clusterconfig/homelab-k8s-k8s-cp-1.yaml
talosctl apply-config --nodes 172.16.20.21 --insecure \
  --file kubernetes/talos/clusterconfig/homelab-k8s-k8s-cp-2.yaml
talosctl apply-config --nodes 172.16.20.22 --insecure \
  --file kubernetes/talos/clusterconfig/homelab-k8s-k8s-cp-3.yaml

# 3. re-bootstrap etcd from scratch (run on ONE node only)
talosctl bootstrap \
  --nodes 172.16.20.20 \
  --endpoints 172.16.20.20 \
  --talosconfig kubernetes/talos/clusterconfig/talosconfig

# 4. get fresh kubeconfig
talosctl kubeconfig \
  --nodes 172.16.20.20 \
  --endpoints 172.16.20.19 \
  --talosconfig kubernetes/talos/clusterconfig/talosconfig

# 5. install Cilium (same command as Phase 9)

# 6. re-bootstrap FluxCD (same command as Phase 10)

# 7. restore Sealed Secrets key (before FluxCD reconciles apps)
sops -d /mnt/backups/keys/sealed-secrets-key.sops.yaml | kubectl apply -f -
kubectl rollout restart deployment sealed-secrets -n kube-system
#    FluxCD reconciles everything; controller uses the restored key
```

### Netbird (VPN) IP leakage into cluster networking

Netbird adds a `wt0` WireGuard interface with a `100.80.x.x/16` address to every node.
Several Kubernetes components auto-detect the "primary" node IP and will pick `100.80.x.x` if not constrained.
The `talconfig.yaml` has three guards to prevent this:

| Component | Guard | Location in talconfig |
|---|---|---|
| etcd | `advertisedSubnets`/`listenSubnets: [172.16.20.0/24]` | `controlPlane.patches` |
| kubelet | `machine.kubelet.nodeIP.validSubnets: [172.16.20.0/24]` | `controlPlane.patches` |
| kube-apiserver | `cluster.apiServer.extraArgs.advertise-address: <LAN IP>` | per-node `patches` |

**Symptom — kube-apiserver endpoints use Netbird IPs:**
containerd uses the Netbird IP as `status.podIP` for hostNetwork pods (it reads the host's IP list and picks `100.80.x.x`). The kube-apiserver static pod has `--advertise-address=$(POD_IP)`, so it registers the Netbird IP in the `kubernetes` service endpoints. Pods that land on a Cilium node that can't route Netbird IPs get DNS/API timeouts.

Check: `kubectl get endpointslices -n default` — endpoints must all be LAN IPs.
Fix: per-node `advertise-address` overrides the `$(POD_IP)` default. Apply config and restart kube-apiserver pods.

**Symptom — CiliumNode has Netbird IP as InternalIP:**
If Cilium registers cp-N with its Netbird IP, cross-node VXLAN tunnels use the wrong destination. Pods on other nodes can't reach endpoints on cp-N.

Check: `kubectl get ciliumnodes -o wide` — all `INTERNALIP` entries must be LAN IPs.
Fix: `kubectl delete ciliumnode <node>` — the Cilium agent recreates it with the Kubernetes node's InternalIP (which kubelet now reports correctly via `nodeIP.validSubnets`).

**Symptom — etcd peers use Netbird IPs:**
etcd cluster loses quorum if peers can't reach each other. One node survives on LAN; others fail.
Fix: `etcd.advertisedSubnets`/`listenSubnets` in `talconfig.yaml` — apply and reboot the affected nodes.
Recovery if quorum is already broken: see `force-new-cluster` notes below.

**etcd force-new-cluster (last resort):**
If etcd has lost quorum, pick the node with the most recent data, add a temporary node-level patch:

```yaml
# in talconfig.yaml nodes[] entry (remove after recovery)
patches:
  - |-
    cluster:
      etcd:
        extraArgs:
          force-new-cluster: "true"
```

Apply to that one node, wait for it to become healthy, then wipe `EPHEMERAL` on the other nodes so they rejoin:

```bash
talosctl reset --system-labels-to-wipe EPHEMERAL \
  --nodes 172.16.20.21,172.16.20.22 \
  --talosconfig talos/clusterconfig/talosconfig \
  --reboot

# After they rejoin, remove force-new-cluster, regenerate, apply to the recovered node
```
