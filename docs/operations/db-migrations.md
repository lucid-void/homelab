# Database Operations & Manual Migrations

Ad-hoc SQL procedures for situations not covered by application tooling — primarily user data transfers and ownership fixups in Immich. Run everything through a psql session on the DB VM.

```bash
ssh user@172.16.20.10
docker exec -it databases_postgres.1.<hash> psql -U immich immich
```

---

## Immich: Transfer assets between users

### When this is needed

- Migrating from a local password account to an OAuth/SSO account
- Consolidating two accounts after an identity provider change

### Scope

The following tables require ownership updates. Order matters — assets first, then dependents.

| Table | Column | Notes |
|---|---|---|
| `asset` | `"ownerId"` | Primary asset ownership |
| `album` | `"ownerId"` | Albums created by the user |
| `person` | `"ownerId"` | Facial recognition people |

Tables that do **not** need updating (they reference by asset/person ID, not user ID):
`asset_face`, `asset_exif`, `album_asset`, `face_search`

### Procedure

Replace the example UUIDs with the real ones from `SELECT id, email FROM "user";`.

```sql
-- Verify source and destination before running
SELECT id, email FROM "user"
WHERE id IN (
  'aaaaaaaa-0000-0000-0000-000000000001',  -- old user
  'bbbbbbbb-0000-0000-0000-000000000002'   -- new user
);

-- 1. Transfer assets
UPDATE asset
SET "ownerId" = 'bbbbbbbb-0000-0000-0000-000000000002'
WHERE "ownerId" = 'aaaaaaaa-0000-0000-0000-000000000001';

-- 2. Transfer albums
UPDATE album
SET "ownerId" = 'bbbbbbbb-0000-0000-0000-000000000002'
WHERE "ownerId" = 'aaaaaaaa-0000-0000-0000-000000000001';

-- 3. Transfer person records (facial recognition people)
UPDATE person
SET "ownerId" = 'bbbbbbbb-0000-0000-0000-000000000002'
WHERE "ownerId" = 'aaaaaaaa-0000-0000-0000-000000000001';
```

### Verify

```sql
SELECT
  (SELECT COUNT(*) FROM asset    WHERE "ownerId" = 'bbbbbbbb-0000-0000-0000-000000000002') AS assets,
  (SELECT COUNT(*) FROM album    WHERE "ownerId" = 'bbbbbbbb-0000-0000-0000-000000000002') AS albums,
  (SELECT COUNT(*) FROM person   WHERE "ownerId" = 'bbbbbbbb-0000-0000-0000-000000000002') AS persons;

-- Old user should have 0 in all columns
SELECT
  (SELECT COUNT(*) FROM asset    WHERE "ownerId" = 'aaaaaaaa-0000-0000-0000-000000000001') AS assets,
  (SELECT COUNT(*) FROM album    WHERE "ownerId" = 'aaaaaaaa-0000-0000-0000-000000000001') AS albums,
  (SELECT COUNT(*) FROM person   WHERE "ownerId" = 'aaaaaaaa-0000-0000-0000-000000000001') AS persons;
```

### Side effects

- All three `UPDATE` statements fire the `updated_at()` trigger, which stamps a fresh `updateId` (UUIDv7) on every modified row.
- The mobile app's next sync will receive all transferred records as if they were newly created — a full re-sync is expected and normal.
- `asset_audit` entries are **not** generated by raw SQL updates. The audit log will show a gap for the transferred assets. This is cosmetic only — all data is intact.

## Lessons learned: the missing `person` transfer

During the initial migration of a local account to an OAuth account, only `asset` and `album` rows were transferred. The `person` table was overlooked.

**Symptom:** Mobile app backup loop — `POST /api/sync/stream` returned ~341 KB on every attempt and the app never reached `POST /api/assets`. App log showed:

```
SqliteException(787): FOREIGN KEY constraint failed
Causing statement: INSERT INTO "asset_face_entity" ... person_id = <uuid>
```

**Root cause:** Immich's sync protocol sends `asset_face` records scoped to the assets' owner, but sends `person` records scoped to the persons' owner. After the partial migration the faces referenced persons still owned by the old account. The app received the face records but not the corresponding person records, so the local SQLite FK constraint failed and the sync aborted before any photos were uploaded.

**Fix:** Run step 3 from the procedure above (transfer `person` records). The app's next sync picks up the persons automatically — no checkpoint clearing or service restart needed.
